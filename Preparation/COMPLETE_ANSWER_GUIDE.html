1. MICROSERVICES & ARCHITECTURE (30 Questions)

Key Context: We have 9 microservices: API Gateway, Auth Service, Quiz Service, Live Service, Result Service, Gamification Service, Admin Service, Social Service, and Moderation Service. Each runs independently with its own database, deployed on Docker.
Q1:Why microservices?

Independent scalability - Live Service peaks during class hours (9-11 AM, 3-5 PM), while Quiz Service peaks at night. With monolith, we'd scale entire app for one peak. Microservices let us scale Live to 10x and Quiz to 3x independently. Technology flexibility - Quiz Service uses Node.js + MongoDB for flexibility, Admin Service uses Express + PostgreSQL for reliability. Team autonomy - two teams can work on different services without coordination. Faster deployment - Quiz Service changes don't affect Live Service, reducing risk and deployment time from hours to minutes.

Q2:How do you define service boundaries?

Using Domain-Driven Design (DDD). Identify subdomains: User Management (registration, profiles), Quiz Management (create, store questions), Live Sessions (real-time interactions), Results Analysis (grading, statistics), Gamification (badges, leaderboards). Each service owns its domain completely. Test: "Can one team own this?" "Are scaling needs different?" "Can this be deployed independently?" If yes to all, it's a good service.

Q3:Microservices trade-offs?

Pros: Scalability (scale what you need), resilience (one service down â‰  everything down), fast deployment, tech flexibility, team autonomy. Cons: Complex debugging (trace request across 5 services), network latency (intra-service calls ~5-50ms vs ~1ms in-process), data consistency (no ACID transactions), operational complexity (monitor 9 services). We accept cons because benefits outweigh.

Q4:How handle distributed transactions?

Saga pattern with event sourcing. Example: User completes quiz. (1) Quiz Service saves attempt, publishes "QuizCompleted" event to Redis Pub/Sub. (2) Result Service listens, calculates score, publishes "ScoreCalculated". (3) Gamification Service listens, awards badges, publishes "BadgeAwarded". (4) Social Service listens, updates stats. Each step is idempotent (running twice gives same result). We accept eventual consistency - badges appear 1-2 seconds later, users don't mind.

Q5:Service discovery?

In development: hardcoded service URLs in .env files. In production: Docker DNS - services reference each other by container name (quiz-service.local, live-service.local). API Gateway maintains service registry - each service sends /health heartbeat every 10 seconds. If service doesn't respond for 30 seconds, marked unhealthy and requests route elsewhere. Simple but effective for our scale.

Q6:API versioning?

URL-based: /api/v1/quizzes, /api/v2/quizzes. Old versions supported 3 months after new release, then deprecated. Maintain backward compatibility where possible - added new fields optional. Use feature flags for gradual rollout. Consumers (frontend, mobile) test against current AND next version during dev phase to prepare for upgrades. Document breaking changes clearly.

2. DATABASE & STORAGE (35 Questions)

Q31:Why MongoDB vs SQL?

Flexible schema - quiz question structures vary (MCQ vs fill-in-the-blank vs essay, different option counts). Relational DB needs schema migrations. MongoDB handles on-the-fly. Document model maps naturally to domain - Quiz is a document, Questions are array. Horizontal scalability - using MongoDB sharding for large collections. Aggregation pipeline perfect for analytics: leaderboard queries, user statistics, cohort analysis. Trade-off: no ACID, but for our data, eventual consistency is acceptable.

Q32:Document schema design?

Embed related data when access patterns are tight (Quiz embeds Questions array). Reference when one-to-many with variable sizes (Users reference Attempts - store only recent 100). Denormalize for read performance (store user stats in user doc, update on score change). Use TTL indexes for temporary data: sessions TTL 30 min, reset tokens TTL 1 hour. Include metadata: createdAt, updatedAt, version for optimistic locking.

Q34:Indexing strategy?

Index fields used in queries and sorts: userId (user's quizzes), createdAt (recent first), subject (filtering). Compound indexes for common patterns: {userId, createdAt} for "user's recent quizzes". Monitor slow queries with MongoDB profiler. Avoid indexing rarely-queried fields (wastes memory). For high-cardinality fields, sparse indexes. Trade-off: indexes speed queries but slow writes and consume memory.

Q35:Prevent N+1 queries?

Aggregate at database level - use MongoDB aggregation pipeline to fetch user + stats in one query. Batch queries - fetch 100 user profiles in one query instead of 100 separate queries. Denormalize - store frequently-accessed data in parent document (user doc includes stats). Eager loading - when fetching Quiz, also fetch related Questions. Redis caching for repeated queries.

3. REAL-TIME & SOCKET.IO (40 Questions)

Q66:Why Socket.IO vs WebRTC vs WebSockets?

Socket.IO handles fallback (WebSocket â†’ XHR polling) for network compatibility. Built-in rooms for live quiz sessions (all students in one room see same questions). Namespace support (separating live, social, notifications). Auto-reconnection when network drops, message buffering. WebRTC is P2P, not suitable for server-to-client push. Raw WebSockets lack all this. Socket.IO fits our needs perfectly.

Q67:Socket.IO at scale?

Using Redis adapter. Instance 1 (1000 connections) publishes to Redis â†’ Redis broadcasts to instances 2,3,4 â†’ each broadcasts to their connections. Prevents duplicate messages. Use Redis Pub/Sub for events crossing multiple server instances. Load balance connections round-robin. During deploys, gracefully drain: route new connections to new servers, existing drain over 30 minutes. Monitor: connection count, message throughput, latency.

Q76:Socket.IO rooms for live quizzes?

Each live quiz session is a room: room "live-session-12345". All students join this room. When teacher publishes question, emit to room - all students see simultaneously. Leaderboard updates emit to room in real-time. When student disconnects, remove from room. When last student leaves, delete room. Rooms are in-memory on each Socket.IO instance, synced via Redis for cross-instance awareness.

4. CACHING & REDIS (30 Questions)

Q106:Why Redis Cloud vs in-memory?

In-memory caching in each app instance creates stale data issues. With 3 server instances, user sets cache in instance 1, next request hits instance 2, sees stale data. Redis Cloud is shared source of truth. Provides: persistence (survives restarts), replication (HA), scalability (beyond single server memory), cross-instance data sharing (critical for Socket.IO adapter), Pub/Sub for messaging.

Q107:What data cached? How long?

Sessions: 30 min User auth state. Leaderboards: 5 min Recalculated periodically for freshness. Quiz questions: 24h Rarely change once published. User profiles: 1h Profile updates are rare. Rate limit counters: 15 min Sliding window for 100 req/15min limit. Live session state: duration + 1h During live quiz + buffer for replay. TTL based on update frequency and freshness requirements.

Q110:Handle cache stampede?

Cache stampede: many requests hit same expired key simultaneously, all hit database. Solution: Probabilistic early expiration (expire at 4.5 min instead of 5 min) - if hit before expiry, refresh. Or use locks: first request locks and regenerates, others wait. Or use stale-while-revalidate: serve stale cache while refreshing in background. We use probabilistic approach - simple and effective.

5. SECURITY & AUTHENTICATION (40 Questions)

Q161:JWT authentication flow?

(1) User logs in: email/password validated by Auth Service OR redirected to Google OAuth. (2) Auth Service returns: access token (15 min expiry) + refresh token (7 days, httpOnly cookie). (3) Client sends access token in Authorization header: "Bearer {token}". (4) API Gateway validates signature using public key, checks expiry. (5) On token expiry, client uses refresh token to get new access token from Auth Service. (6) Refresh token stored in httpOnly, secure, sameSite=strict cookies - safe from XSS.

Q162:Access token validity?

15 minutes. Short duration limits exposure if token is compromised - attacker has 15 minute window. Users don't notice expiry because refresh is automatic. Refresh tokens (7 days) provide longer access without storing passwords. Common approach: Google, GitHub use similar 15-60 min access tokens.

Q163:Refresh token handling?

Stored in httpOnly, secure, sameSite=strict cookies (inaccessible to JavaScript). When access token expires, client calls GET /auth/refresh, sends refresh token in cookie (automatically included). Server validates refresh token (hasn't been revoked, not expired), issues new access token. Refresh token doesn't rotate for simplicity, but we store token issue time and issue new one if too old (7 days).

Q181:Password hashing?

bcrypt with 12 rounds (recommended: 10-12). Each password salted during hashing. Even if database is compromised, hashes are useless. Passwords never stored in logs. Never sent in URLs or query params (POST only). Reset flow: user clicks "forgot password" â†’ Email Service sends reset link â†’ link has 1-hour expiry token â†’ user sets new password â†’ old refresh tokens invalidated immediately.

Q195:Prevent SQL injection?

We use MongoDB (NoSQL), not SQL. NoSQL injection still possible if inputs aren't validated. We validate/sanitize all inputs using joi schema validation. Example: quiz title must be string 1-100 chars. Store procedures aren't applicable. Parameterized queries equivalent in MongoDB: use aggregation framework instead of building queries with string concatenation.
