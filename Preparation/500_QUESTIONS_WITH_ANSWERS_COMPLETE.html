üèóÔ∏è Microservices Design

1.Why did you choose microservices over monolithic architecture? Medium
Answer: We chose microservices to achieve independent scalability, team ownership, and technology flexibility. Each service (Quiz, Live, Gamification, etc.) can scale based on its own demand patterns. This allows teams to deploy independently, use appropriate tech stacks, and reduce blast radius of failures. For an EdTech platform with multiple feature areas, this is crucial for agility and faster iteration.
2.How do you define service boundaries? What's the criteria? Hard
Answer: We use domain-driven design (DDD) principles based on business capabilities: Quiz Management, Live Sessions, Results Analysis, Gamification, User Management, etc. Each service has clear responsibilities, minimal coupling, and can be owned by one team. We ask: "Does this service have distinct scaling needs?" and "Can this be deployed independently?" If both answers are yes, it's a good boundary.
3.What are the trade-offs of microservices architecture? Hard
Answer: Pros: Independent scaling, team autonomy, technology flexibility, better resilience (one service down doesn't bring all down). Cons: Complex distributed debugging, network latency, eventual consistency challenges, operational complexity. We accept these trade-offs because our traffic patterns vary‚ÄîLive Service peaks during school hours, Quiz Service during evenings, Social Service during breaks.
4.How do you handle distributed transactions across services? Hard
Answer: We use the Saga pattern with event sourcing. When a user completes a quiz: (1) Quiz Service saves result, (2) publishes QuizCompleted event to Redis Pub/Sub, (3) Result Service updates scores, (4) Gamification Service awards badges, (5) Social Service updates stats. Each step is idempotent. We accept eventual consistency (1-2 second delays for badges) as acceptable for UX.
5.What's your service discovery mechanism? Medium
Answer: In development, services use hardcoded URLs. In production, we use Docker DNS‚Äîcontainer names resolve automatically. The API Gateway maintains a registry of healthy service instances using /health endpoints and implements retry logic with exponential backoff. We also use environment variables to configure service URLs, making it easy to update without redeployment.
6.How do you version your APIs across services? Medium
Answer: We use URL versioning (/api/v1/, /api/v2/). Old API versions remain supported for 3 months after new version release. We maintain backwards compatibility where possible. Services document breaking changes in their API changelogs. Consumer services test against both current and next version during development to prepare for upgrades.

üíæ Database & Storage (Sample Answers)

31.Why did you choose MongoDB over SQL databases? Medium
Answer: MongoDB provides flexible schema which is crucial for EdTech‚Äîquiz question structures vary, user attributes change. We needed horizontal scalability for quiz data. The document model maps naturally to our data: Quiz = document, Questions = array, Answers = nested objects. MongoDB's aggregation pipeline is perfect for analytics queries (leaderboards, statistics).
32.How do you design your document schema? Medium
Answer: We embed related data when access patterns are tight (Quiz embeds Questions). We reference when relationships are one-to-many with variable sizes (Users reference Attempts). Denormalize frequently accessed aggregates (store user stats in user doc). Use TTL indexes for temporary data (sessions). Always include metadata: createdAt, updatedAt, version for optimistic locking.

‚ö° Real-Time & Socket.IO (Sample Answers)

66.Why Socket.IO instead of WebRTC or raw WebSockets? Medium
Answer: Socket.IO handles fallback (WebSocket ‚Üí XHR polling) for network compatibility. Built-in room management perfect for live quiz sessions. Namespace support for organizing features. Auto-reconnection and message buffering handle network hiccups. Broadcasting to multiple clients is simpler than raw WebSockets. We need server-to-client push (leaderboard updates, new questions) which WebRTC doesn't provide.
67.How do you handle Socket.IO connections at scale? Hard
Answer: Using Redis adapter: Socket.IO instance 1 publishes to Redis ‚Üí Redis broadcasts to all instances ‚Üí each instance broadcasts to its connected clients. Prevents duplicate messages. Use Redis Pub/Sub for global events. Load balance connections across multiple Node servers. Monitor connection count, disconnect gracefully during deploys using /admin endpoint to reduce spike.

üîÑ Caching & Redis (Sample Answers)

106.Why Redis Cloud instead of in-memory caching? Medium
Answer: Redis Cloud provides: (1) Persistence for reliability, (2) Replication for HA, (3) Scalability beyond single server memory, (4) Cross-instance data sharing (critical for Socket.IO adapter), (5) PubSub for inter-service communication, (6) Built-in backups. In-memory caching in each app instance creates stale data issues. Redis is single source of truth for sessions, locks, and real-time data.
107.What data do you cache? How long? Hard
Answer: Sessions (30 min): User auth state. Leaderboards (5 min): Recalculated periodically to stay fresh. Quiz questions (24h): Rarely change once published. User profiles (1h): Basic info. Rate limit counters (15min): For 100 req/15min limit. Live session state (duration + 1h): During live quiz + buffer for replay. TTL is set based on update frequency and freshness requirements.

üîí Security & Authentication (Sample Answers)

161.Explain your JWT authentication flow? Medium
Answer: (1) User logs in with email/password or Google OAuth, (2) Auth Service validates & returns access token (15 min expiry) + refresh token (7 days, httpOnly cookie), (3) Client sends access token in Authorization header, (4) API Gateway validates signature & expiry, (5) On expiry, client uses refresh token to get new access token. Access tokens are short-lived for security; refresh tokens are long-lived but secure in httpOnly cookies.
162.How long are access tokens valid? Easy
Answer: 15 minutes. Short duration limits exposure if token is compromised. Users don't notice since refresh is automatic via refresh token. If token is leaked, attacker has only 15 minutes window. Refresh tokens (7 days) are stored in httpOnly cookies and never exposed to JavaScript, preventing XSS attacks from stealing them.
