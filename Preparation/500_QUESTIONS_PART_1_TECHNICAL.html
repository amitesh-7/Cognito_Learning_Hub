Microservices Design

ARCHITECTURE
1.Why did you choose microservices over monolithic architecture? Medium
Answer: We chose microservices to achieve independent scalability, team ownership, and technology flexibility. Each service (Quiz, Live, Gamification, etc.) can scale based on its own demand patterns. This allows teams to deploy independently, use appropriate tech stacks, and reduce blast radius of failures. For an EdTech platform with multiple feature areas, this is crucial for agility.
2.How do you define service boundaries? What's the criteria? Hard
Answer: We use domain-driven design (DDD) principles. Boundaries are defined by business capabilities: user management, quiz creation, live sessions, results analysis, etc. Each service has clear responsibilities and minimal coupling. We ask: "Can this service be owned by one team?" and "Does this service have distinct scaling needs?" If yes to both, it's a good boundary.
3.What are the trade-offs of microservices architecture? Hard
Answer: Pros: Independent scaling, team autonomy, technology flexibility, resilience. Cons: Complex distributed debugging, network latency, data consistency challenges, operational overhead. We accept these trade-offs because our traffic patterns vary significantly across services‚ÄîLive Service peaks during class hours, Quiz Service during evenings.
4.How do you handle distributed transactions across services? Hard
Answer: We use the Saga pattern with Redis-backed event sourcing. When a user completes a quiz: Quiz Service saves result ‚Üí publishes event ‚Üí Result Service updates scores ‚Üí Gamification Service awards badges. Each step is idempotent. We accept eventual consistency because users don't mind a 1-2 second delay for badges to appear.
5.What's your service discovery mechanism? Medium
Answer: We use environment variables + service mesh approach. In development, services are hardcoded. In production, we use Docker compose service names which DNS resolves. For advanced scenarios, we've configured a simple load balancer at the API Gateway that maintains a registry of healthy service instances using health checks.
